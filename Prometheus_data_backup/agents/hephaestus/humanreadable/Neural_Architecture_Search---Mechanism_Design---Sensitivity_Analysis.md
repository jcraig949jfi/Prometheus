**Algorithm: NAS‑guided Logical Form Search with Incentive‑Compatible Scoring and Perturbation‑Based Sensitivity**

1. **Data structures**  
   - `PromptTokens`: list of strings from regex‑extracted lexical items (negations, comparatives, conditionals, numbers, causal verbs, ordering tokens).  
   - `AnswerHypotheses`: list of candidate parses, each a directed acyclic graph (DAG) `G = (V, E)`. Nodes `V` are atomic propositions (e.g., “X > Y”, “¬P”, “cause(A,B)”). Edges `E` encode logical connectives (AND, OR, IMPLIES) derived from a fixed grammar.  
   - `ArchitecturePool`: a set of DAG topologies generated by a simple NAS loop (mutation: add/remove node, swap edge type; crossover: splice sub‑graphs). Each topology carries a weight vector `w ∈ ℝ^k` (performance predictor) initialized to zeros.  
   - `ScoreTable`: numpy array shape `(n_candidates, n_architectures)` storing raw scores.

2. **Operations**  
   - **Parsing stage**: Run a deterministic regex‑based extractor over the prompt to fill `PromptTokens`.  
   - **NAS search**: For `T` iterations, mutate/crossover topologies in `ArchitecturePool`. For each topology `a`, compute a *performance predictor* `p_a = f_w(PromptTokens)` where `f_w` is a linear model `w·φ(PromptTokens)` with feature map `φ` counting occurrences of each token type (negation, comparative, etc.). Update `w` via ridge regression on the current batch of candidate answers (standard library `numpy.linalg.lstsq`).  
   - **Mechanism‑design scoring**: For each candidate answer `c` and architecture `a`, evaluate logical consistency: run a forward‑chaining constraint propagator (modus ponens, transitivity) on the DAG instantiated with `c`’s propositions. If a contradiction is found, assign penalty `‑∞`; otherwise compute a *utility* `u_{c,a} = –|Violations|`. Apply the Vickrey‑Clarke‑Groves (VCG) payment rule: the score for `c` under `a` is `s_{c,a} = u_{c,a} – Σ_{j≠c} u_{j,a}`. This makes truthful reporting a dominant strategy.  
   - **Sensitivity analysis**: Perturb `PromptTokens` by randomly flipping a token’s polarity (negation ↔ affirmation) or numeric value ±ε (ε drawn from N(0,0.1)). Re‑compute `s_{c,a}` for each perturbation; the final score for `c` under `a` is the mean over `P` perturbations. Aggregate across architectures by weighted sum using the latest `w` (higher‑performing architectures get higher weight). The resulting scalar is the candidate’s overall score.

3. **Structural features parsed**  
   - Negation tokens (“not”, “no”), comparatives (“greater than”, “less than”), conditionals (“if … then”), causal verbs (“cause”, “lead to”), numeric values, ordering relations (“before”, “after”), and conjunction/disjunction markers.
