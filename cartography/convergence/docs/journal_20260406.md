### Architecture Evolution (chronological)

1. **MVP** — council_client + search_engine + thread_tracker + research_cycle
   - 4 files. LLM generates, LLM evaluates, report.
   - Problem: LLM evaluation is narrative-building, not falsification.

2. **Falsification battery** — 11 computational kill tests
   - Adapted from Charon's murder board pattern.
   - Permutation null, subset stability, effect size, confound sweep,
     normalization sign-flip (THE test that killed Charon's April 5 narrative),
     Bonferroni, dose-response, direction consistency, simpler explanation,
     outlier sensitivity, cross-validation.
   - Battery overrides LLM. KILLED = falsified. Period.

3. **Structured logging** — JSONL dual output
   - Every API call, search, battery test, thread transition logged with
     timestamps, token counts, elapsed times, cumulative totals.

4. **Council review** — all 4 providers critique the pipeline
   - Unanimous finding: hypothesis quality is weakest link, battery is strongest.
   - Concrete suggestions: validate search plans, check evidence relevance,
     pre-battery data quality gate. All implemented same session.

5. **Validation gates** (from council review)
   - Hypothesis gate: dry-run search plan, inspect.signature param validation
   - Evidence gate: terminate threads where all searches fail
   - Pre-battery gate: min sample size, zero variance, NaN/Inf check
   - Thread termination: search_failed, irrelevant_evidence statuses

6. **LLM evaluation removed** — battery is authoritative
   - Saved ~6s + ~200 tokens per thread.
   - Pipeline now uses 1 LLM call per cycle (hypothesis generation only).

7. **Aggregate searches** — the unlock
   - lmfdb_rank_comparison: returns 30K+ paired conductor values
   - lmfdb_conductor_distribution: histograms by rank
   - oeis_by_id: fetch specific A-numbers
   - oeis_find_containing: coverage scoring
   - Battery extraction updated to recognize paired arrays.
   - First battery run on real data: 15,599 vs 14,783 conductors.

8. **Computational branching** — no LLM
   - KILLED by effect size → refine (finer bins)
   - KILLED by normalization → pivot (test log-transform)
   - KILLED by permutation → adjacent parameter
   - SURVIVES → cross-family generalization
   - SKIP → data refinement
   - Branches feed as seeds to next cycle.

9. **Loop mode** — `--loop N`
   - Branches from cycle K become seeds for cycle K+1.
   - First SURVIVE: rank-1 vs rank-0 conductor distributions at 1000-2000 range.
   - 3 nested loops: inner (research), middle (tensor review), outer (council review).

10. **Random dataset selection** — pick 4 of N per cycle
    - Keeps prompts tight, forces novel combinations, scales with new datasets.

11. **Tensor review** — computational dataset quality audit
    - 5 datasets reviewed in 3 seconds. 12 suggestions auto-filed.
    - Key findings: OEIS names corrupted, Materials fields 0% complete.

12. **External research feed** — daily

    - Gemini synthesis (optional, counts against daily limit)

13. **Suggestions ledger** — HITL gate

### The Finding That Validated The Pipeline

Thread: "Rank-0 vs rank-1 conductor distributions differ"
- Battery ran on 15,599 rank-0 + 14,783 rank-1 real conductor values
- Cohen's d = -0.16 (below small effect threshold)
- Cross-validation accuracy = 52.9% (barely above chance)
- Scale warning: mean-normalization kills the effect
- **KILLED.** Conductor distributions do NOT meaningfully differ by rank.
- This is the same kind of result Charon found on April 5: what looks like
  a difference is really a trivial scale effect.

Thread: "Rank-1 vs rank-0 at conductor 1000-2000 with bin counts"
- Battery: 8 PASS, 0 FAIL, 3 SKIP — **SURVIVES**
- Auto-branched to cross_family (test on modular forms)
- Needs HITL review before "confirmed"
