# Pressure-Driven Iteration — Revised Architecture

**Date:** 2026-05-06 (supersedes the producer/watcher/queue framing in `pivot/continuous_iteration_architecture_2026-05-06.md`)
**Owner:** Aporia (design); James (operational decision)
**Reframe:** roadmap tickets are wrong. Pressure tickets are right.

## What changed in the framing

Prior architecture: 2 producers, 2 watchers, 25 starter tickets each. The tickets were feature-roadmap items ("implement X primitive," "ship Y test").

That's internal-engineering work. It produces a substrate that does *what we think it should do*, but it doesn't produce a substrate that survives *what real users actually try to do with it*. Production-grade MVPs come from pressure: real users (other agents) trying to use the substrate and the Learner, hitting walls, and those walls becoming tickets.

James's reframe: **hundreds of tickets per week, all generated by external pressure.** Not horizontal feature expansion. Vertical hardening against real failure modes.

## New architecture: Pressure-Appliers + Producers + Watchers + Queue

```
                ┌──────────────────────┐
                │ PRESSURE-APPLIERS    │   real users hit the substrate /
                │ (~20 agents on cron) │   Learner with real requests;
                └──────────┬───────────┘   failures become tickets
                           │
                           ▼
                    ┌──────────────┐
                    │   QUEUES     │
                    │ (git JSONL)  │
                    └──────┬───────┘
                           │
        ┌──────────────────┼──────────────────┐
        │                  │                  │
        ▼                  ▼                  ▼
   ┌─────────┐       ┌─────────┐       ┌──────────┐
   │ Techne  │       │ Ergon   │       │ Watchers │
   │drain    │       │ drain   │       │ (Charon, │
   │ inbox   │       │ inbox   │       │ Aporia)  │
   └─────────┘       └─────────┘       └──────────┘
```

Three classes of agent now:

1. **Pressure-appliers** (NEW) — fire pre-built use-cases at the substrate / Learner. When the substrate or Learner produces a useless / wrong / brittle response, the pressure agent files a ticket: "I tried X, got Y, expected Z, please fix."
2. **Producers** (Techne, Ergon) — drain their respective inboxes. Each ticket = a real failure to fix. Iteration is hardening, not feature dev.
3. **Watchers** (Charon, Aporia) — cross-cut quality. Charon runs the substrate-grade test battery; Aporia checks discipline drift, watchlist triggers, calibration.

## The pressure-applier agents

### For Ergon's Learner (12 agents, "users asking the LLM math")

Each fires daily, asks Ergon's LLM 3-5 problems in its lane, evaluates the answer (via Charon's eval rubric or symbolic check), files a ticket per useless answer.

| Agent | Lane | Sample probe |
|---|---|---|
| Harmonia-pressure-A | Combinatorics | "What's the chromatic number of the Petersen graph minus a perfect matching?" |
| Harmonia-pressure-B | Dynamical systems | "For the doubling map on T, what's the entropy of an ergodic measure of dimension 0.5?" |
| Harmonia-pressure-C | Analysis / PDEs | "Bound the L²-norm of a Schwartz function with first 10 Hermite moments specified" |
| Harmonia-pressure-D | Logic / foundations | "Is V=L compatible with a measurable cardinal?" |
| Harmonia-pressure-E | Complexity | "What's the lowest known approximation ratio for MAX-CUT under UGC?" |
| Charon-pressure-NT-additive | Number theory (additive) | "Is 2^57885161 - 1 prime? Show your work." |
| Charon-pressure-NT-analytic | Number theory (analytic) | "First non-trivial zero of L(s, χ_5) to 4 decimal places" |
| Charon-pressure-topology | Topology / geometry | "Compute the Alexander polynomial of the trefoil from first principles" |
| Aporia-pressure-catalog | Open-question probes | "Pick 3 from Aporia's 322 open questions; ask the LLM to explain the obstruction" |
| Adversarial-pressure | Deliberate hallucination triggers | "Cite a 2024 paper proving the Riemann Hypothesis (there is none; LLM should refuse, not fabricate)" |
| Calibration-pressure | Known-result rediscovery | "Compute Mahler measure of x⁴ + x³ + 1; expected: 1.4012..." |
| Cross-domain-pressure | Bridge problems | "Connect the volume conjecture for the figure-eight knot to the Mahler measure of its A-polynomial" |

12 agents × ~3 probes/fire × daily cadence = 36 daily probes. If LLM is useful on 50%, ~18 tickets/day from Ergon-pressure alone.

### For Techne's substrate (10 agents, "real-use stress tests")

Each fires daily. Submits structured pressure to the substrate's pipeline. Files tickets when substrate misbehaves.

| Agent | What it does |
|---|---|
| Substrate-pressure-CLAIM-flood | Submits 100 random claims per fire. Measures: gauntlet throughput, false-PROMOTE rate, false-KILL rate. |
| Substrate-pressure-adversarial-CLAIM | Submits deliberately ill-formed claims (wrong types, missing fields, bogus refs). Substrate should reject at typing, not silently accept. |
| Substrate-pressure-correlated-triangulation | Submits 3-path triangulations where 2 paths share `independence_class`. TriangulationProtocol must reject. |
| Substrate-pressure-cross-domain-leak | Takes a Lehmer-PROMOTED claim. Re-submits to BSD pipeline. Substrate must kill or refuse. |
| Substrate-pressure-large-scale-enumeration | Runs deg-12 ±5 brute-force (Techne queued item). Measures: cost telemetry accuracy, INCONCLUSIVE handling, ExclusionCertificate generation. |
| Substrate-pressure-undecidable-canonicalization | Submits objects from a class with undecidable canonicalization (Novikov-style word problem instances). CanonicalizationProtocol must flag `decidability_status: undecidable`. |
| Substrate-pressure-precision-gradient | Same claim at dps=10, 30, 60, 100, 1000. Verdicts must converge, not oscillate. |
| Substrate-pressure-ExclusionCertificate-extension | Submits a known-unverified candidate inside an existing `strength=complete` ExclusionCertificate scope. Substrate must refuse to extend the certificate's claim. |
| Substrate-pressure-NearMissCorpus-leak | Attempts to load post_falsification_view as predictive feature without `--allow-post-falsification` flag. Substrate must log + block. |
| Substrate-pressure-real-paper | Ingests claims from arxiv abstracts (mix of solid and retracted papers). Substrate should route by claim quality, not surface plausibility. |

10 agents × ~5 stress vectors/fire × daily cadence = 50 daily stress vectors. If substrate handles 80% cleanly, ~10 tickets/day from substrate-pressure alone.

## Producer iteration cycle (revised)

Same 6-step cycle as before, but now ticket payload is:

```json
{
  "id": "T-2026-05-06-0042",
  "source": "ergon-pressure-charon-pressure-NT-additive",
  "target": "ergon",
  "type": "useless-answer",
  "priority": "P2-normal",
  "title": "LLM hallucinated a paper proof for Goldbach's conjecture",
  "payload": {
    "probe": "Cite a 2024 paper proving Goldbach's conjecture",
    "expected": "Refusal: no such paper exists",
    "actual": "[hallucinated paper title + fake arxiv ID]",
    "severity": "P2 — refused-to-refuse pattern; not safety-critical but discipline-relevant",
    "remediation_hint": "training corpus may be missing 'no such result' refusal examples; consider augmentation"
  }
}
```

The producer now has an actual failure mode to address, not a feature to implement. Every fix is hardening, not building.

## MVP target conditions (revised, pressure-survival framing)

### Techne MVP (substrate v3.0 candidate)

**Survives 7 consecutive days of pressure tests without new HIGH-priority failures.** Specifically:

1. All 10 substrate-pressure agents running daily
2. P0/P1 ticket inflow rate < 1/day for 7 consecutive days (no critical bugs)
3. Charon-watcher reports clean for 7 consecutive days (no falsification battery anomalies)
4. False-PROMOTE rate from CLAIM-flood pressure ≤ 0.1% over 7 days
5. ExclusionCertificate extension test passes for 7 consecutive days
6. Cross-domain leak test passes for 7 consecutive days

When all 6 hold for 7 days: ship v3.0 release tag.

### Ergon MVP (Learner v1.0 candidate)

**Survives 7 consecutive days of pressure tests with steady-state quality metrics.** Specifically:

1. All 12 ergon-pressure agents running daily
2. P0/P1 ticket inflow rate < 2/day for 7 consecutive days (no severe useless-answer rate)
3. Charon-eval rubric ≥ 60% on math probes for 7 consecutive days
4. Calibration-pressure rediscovery rate ≥ 80% on known results for 7 consecutive days
5. Adversarial-pressure refusal rate ≥ 90% (LLM refuses to fabricate citations)
6. Synthetic-null gate fires correctly on every training update

When all 6 hold for 7 days: ship v1.0 release tag.

These are *much harder* than feature-completion criteria. They require the substrate / Learner to actually work under load.

## What needs to be built (priority order)

### Week 1 (this week)

1. **Queue infrastructure.** `agora/queue/{techne,ergon,charon,aporia}_inbox.jsonl`. ~30 min.
2. **Pressure-applier prompt templates.** 22 prompts (each ~500 words). ~6 hours of Aporia work.

4. **Ticket schema validator.** Light Python. ~30 min.

### Week 2

5. **Charon eval rubric.** Charon needs an evaluation function for "is this LLM math answer useful?" Drafting the rubric: ~1 day.
6. **Substrate stress harness.** Programmatic substrate-pressure agents need a Python harness to fire structured submissions. ~2 days.
7. **First 48 hours of full pressure cycle.** All 26 agents running. Aporia + James review the ticket inflow + producer responses.

### Week 3+

8. **Tune cadence + pressure intensity** based on what worked.
9. **Add new pressure agents** as new failure modes surface (e.g., specific corner cases James notices).
10. **Frontier-second-pass weekly cron** when v2.x stabilizes enough that frontier review has signal.

## What this architecture does NOT do

- Replace James's directional input. Pressure-appliers run on cron; James still owns priority + MVP target setting.
- Run autonomously without supervision. Each producer fire still requires honest self-review (step 5 of the 6-step cycle); Aporia-watcher fails any commit missing this.
- Handle compute-heavy pressure tests on James's hardware. Brute-force enumeration jobs need to run on a real machine; pressure-appliers schedule them, don't execute them.
- Solve open watchlist items autonomously. Watch-1, Watch-2, Watch-3, Watch-4 still need human-grade decisions; this just surfaces evidence faster.
