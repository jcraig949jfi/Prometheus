# Dashboard Deployment Plan — 2026-05-15

**Goal:** Serve `dashboard/index.html` publicly so James can check Prometheus status from his phone or any browser, with auto-refreshing data on a 1-hour cadence.

**Author:** Aletheia, after Track-2 React UI shipped locally.

---

## The architectural problem

The dashboard reads two data files at runtime via `fetch`:
- `state.json` — Agora snapshot regenerated by `scripts/portfolio_monitor.py` every cycle
- `portfolio_brief.md` — LLM brief regenerated by `scripts/metis_portfolio.py` every cycle

Both are gitignored (added to `.gitignore` deliberately to avoid commit-spam on `main`). For GitHub Pages to serve the dashboard, both files have to be reachable via HTTP from the same path as `index.html`. The dashboard fetches them as relative URLs.

This creates a tension:
- **Don't commit data to main** — every hour produces a "portfolio update" commit, polluting git log and triggering CI / unrelated automation.
- **But Pages can only serve files that are in some branch.**

Three viable architectures, ordered from least-to-most operational overhead.

---

## Option A — Auto-commit to `gh-pages` branch (recommended)

**Shape:** `main` stays clean. A separate `gh-pages` branch carries the live dashboard data. The intelligence loop on M4 (or wherever it runs) pushes data updates to `gh-pages` only. GitHub Pages serves `gh-pages` at `https://jcraig949jfi.github.io/Prometheus/`.

**One-time setup (you do in GitHub UI + local):**

1. **Create the gh-pages branch** with the dashboard sources but no main-branch baggage:
   ```powershell
   git checkout --orphan gh-pages
   git rm -rf .
   git checkout main -- dashboard/index.html
   mv dashboard/index.html ./index.html
   echo '{}' > state.json
   echo '_Loading…_' > portfolio_brief.md
   echo '# gh-pages branch — auto-deployed dashboard' > README.md
   git add index.html state.json portfolio_brief.md README.md
   git commit -m "Initial gh-pages: dashboard scaffolding"
   git push origin gh-pages
   git checkout main
   ```

2. **Enable Pages** in repo Settings → Pages → Source: `Deploy from a branch`, Branch: `gh-pages`, Folder: `/ (root)`. Save. Pages will report the public URL.

**Recurring deployment (the intelligence loop does it automatically):**

The loop calls a new sibling script `scripts/push_dashboard.py` after each successful `portfolio_monitor + metis_portfolio` cycle. The script:

1. Uses `git worktree add` to mount the `gh-pages` branch in a sibling directory (e.g., `../Prometheus-ghpages`) — avoids switching branches in the main working tree.
2. Copies the freshly generated `dashboard/state.json` and `dashboard/portfolio_brief.md` over the worktree's copies.
3. `git add . && git commit -m "auto: portfolio update <ISO>"` in the worktree.
4. `git push origin gh-pages` from the worktree.
5. The worktree stays put — no checkout switching, no risk to `main`.

**Trade-offs:**

- ✓ Public access via the GitHub-hosted URL; works from anywhere.
- ✓ `main` stays clean.
- ✓ gh-pages commits are expected to be auto-generated; nobody looks at that log.
- ✗ Hourly push generates ~24 commits/day on `gh-pages`. After 6 months that's ~4K commits — fine for a branch nobody reviews.
- ✗ Public URL exposes Agora state (agent names, stream subjects, work-queue depth). Probably fine for this project; flag if you don't want it public.
- ✗ Requires git push credentials. The intelligence loop is on M4 with your local credentials — works fine while running on your machine; doesn't work if you ever run the loop on a hosted runner without configured credentials.

**Cost:** ~30 min to wire `scripts/push_dashboard.py` + the one-time gh-pages setup.

## Option B — Local server only (current state, no extra work)

**Shape:** `python -m http.server 8000 -d dashboard/` on M4, opened at `http://localhost:8000/` from any device on your LAN (visit `http://<M4-LAN-IP>:8000/` from your phone).

**Trade-offs:**

- ✓ Zero new code. Already works today.
- ✓ No data leaves your network.
- ✗ Only accessible when M4 is on and the server is running.
- ✗ LAN-only — no remote access unless you're on the same network or have a VPN home.
- ✗ Requires you to remember to start the server (or wrap it in a service / Task Scheduler entry).

**Wrap-as-a-service if you stay with this option:**

Add a Windows Task Scheduler entry that runs `python -m http.server 8000 -d C:\Prometheus\dashboard` on user logon. ~5 min.

---

## Option C — Use `/docs` folder on `main`

**Shape:** GitHub Pages can serve from `main`'s `/docs` folder directly. Rename `dashboard/` → `docs/`, then enable Pages from `main /docs`.

**Trade-offs:**

- ✓ Single-branch setup. Simpler than gh-pages.
- ✗ Hourly auto-commits to `main` if you want fresh data. Commit-spam mixes with real engineering history. Filtering via `git log --grep` works but is friction.
- ✗ Triggers any CI on `main` that watches for commits. If you ever add a workflow, it fires 24x/day for nothing.
- ✗ Renaming `dashboard/` → `docs/` is awkward — `docs/` typically means documentation in convention.

**Variant:** keep `dashboard/` but DON'T auto-commit. Manually push when you want a snapshot. Defeats the purpose of "auto-refresh every hour."

---

## Recommendation

**Go with Option A.** The gh-pages branch is the canonical pattern for this exact problem; auto-commit spam stays off `main`; the public URL works from your phone. ~30 min to implement once you decide.

If you want zero new code today: **Option B is already functional** — just `python -m http.server 8000 -d dashboard/` on M4 and bookmark `http://<M4-LAN-IP>:8000/` on your phone (works when you're on the same WiFi).

## Out-of-scope for v0

- **HTTPS for Option B.** GitHub Pages handles this automatically; Option B is HTTP-only on the LAN.
- **Authentication.** The dashboard is read-only and exposes operational state, not credentials. If you want it private, make the repo private (Pages on private repos requires GitHub Pro/Team) or fall back to Option B + VPN.
- **Mobile-app shell.** The dashboard is responsive Tailwind; phones get an OK experience already. A PWA (`manifest.json` + service worker) would let you "install" it to the home screen, but that's polish, not function.
