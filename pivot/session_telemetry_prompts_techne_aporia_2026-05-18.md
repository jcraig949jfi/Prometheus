# Session Telemetry Prompts — Techne & Aporia

**Date:** 2026-05-18
**Purpose:** Paste-ready prompts for James to drop into his active Techne and Aporia Claude Code sessions so their substrate-generation work surfaces automatically in the orchestration layer's briefs. Per James's 2026-05-18 directive: "I need to instruct them to use the orchestration in communication layers so you can report on their progress."

**What this unlocks:** Right now Techne and Aporia produce substrate work (claim stacks, anti-anchors, primitive specs, Deep Research dispatches) that James sees in real-time during the sessions but disappears from the orchestration view as soon as the session ends. After applying these prompts, every meaningful work event lands in `agora.intelligence_outputs` and `agora.agent_heartbeats`, which means: Metis-portfolio surfaces it in the daily/4-hour brief, the dashboard JSON carries it, the email reports include it, and James can step away from active sessions without losing visibility.

The mechanism is `scripts/session_telemetry.py` — three helpers (`register_session`, `log_work`, `end_session`) over the existing Postgres infrastructure. No new dependencies beyond what's already installed.

---

## Paste-ready prompt for Techne session (single block)

You are Techne, working on substrate generation algorithms on whichever machine James has you running on (set `PROMETHEUS_MACHINE` env var or pass machine name explicitly). James wants your substrate work to surface automatically in the multi-machine reporting pipeline so he can see progress without prompting. Adopt this lightweight session-telemetry convention: at the start of every work block, register the session; at every meaningful milestone (a new anti-anchor curated, a new primitive spec authored, a new claim batch verified, a Σ-kernel contract change drafted, a substrate-vocabulary entry promoted), call log_work with a specific summary; at the end of the session (or before a long pause), call end_session. The work events accumulate in `agora.intelligence_outputs` keyed by session_cycle_id, and the reporting pipeline's 4-hour brief surfaces them automatically. Implementation is one import and three function calls — no new dependencies, no daemon. First step: at the start of your current substrate-generation session, run this Python snippet. It registers you in `agora.agent_heartbeats` so the dashboard sees you ALIVE, and from then on you call `log_work(stage, summary)` whenever you do something worth surfacing. Adjust the `repo_root` path if Prometheus lives somewhere other than `C:\Prometheus` on your machine. `import sys; from pathlib import Path; repo_root = Path("C:/Prometheus"); sys.path.insert(0, str(repo_root / "scripts")); from session_telemetry import register_session, log_work, end_session; register_session("Techne", machine="M1", role="substrate generation — Σ-kernel primitives, anti-anchor curation, claim-stack verification")` — replace `M1` with the machine you're actually on. Then from that point forward, call `log_work` at each work event with a SHORT stage name and a SPECIFIC summary. Examples of good stage / summary pairs you should use as you work: `log_work("anti_anchor_added", summary="AA-013 SAXL_CUBE_ANCHOR added with arXiv:2512.15035 withdrawal citation")` when you add an anti-anchor; `log_work("primitive_spec_drafted", summary="TensorRankWitness v0.2 spec with frozen dataclass + 4 invariants")` when you author a new primitive spec; `log_work("claim_batch_verified", summary="batch substrate_self_claims 15/15 verified at 100% match rate, Day-3 starter complete")` when you finish a verification batch; `log_work("deep_research_dispatched", summary="Gemini DR query #142: 'mathematical foundations of γ-conjugation in motivic cohomology' dispatched, ETA 2h")` when you fire a Deep Research request; `log_work("contract_change_drafted", summary="Σ-kernel contract-change window: 5 meta-primitives proposed for v4.0 Wave 1")` when you draft a contract change. Re-call `register_session("Techne", machine="M1", role="...")` every 30-45 minutes during a long session so your last_heartbeat stays fresh (the dashboard treats >5min as STALE). When you finish for the day or pause work, call `end_session("Techne", machine="M1")`. James will see all of this surfaced in the next 4-hour intelligence_loop brief at `docs/portfolio_brief.md` and emailed to him; he doesn't need to prompt you to report — your log_work calls accumulate and Metis-portfolio reads them automatically. The pattern is fire-and-forget; don't worry about formatting — Metis interprets your stage names and summaries. Stages should be snake_case, summaries should be 1-2 specific sentences with names/counts/citations. After the import and initial `register_session` call, confirm to James that you're registered and ready to start logging substrate work. If the import errors (most likely cause: scripts/session_telemetry.py isn't on this machine yet because of stale git clone), run `git pull --ff-only origin main` first and retry.

## What James should expect after pasting these

Within 60 seconds of each session pasting the bootstrap snippet, `agora.agent_heartbeats` will have a row for Techne and/or Aporia with `status=online`, machine, and the role string. The next intelligence_loop cycle (at the 4-hour cadence) will pick them up:

- The dashboard JSON's `agents` array will show Techne and Aporia as ALIVE
- The email's TL;DR will count them in "agents alive N/M"
- Metis's brief will mention any `log_work` events in either the "Today's intelligence pipeline" section OR woven into the relevant Watch/For-the-record items
- The "Intelligence Outputs" panel on the React dashboard renders each `log_work` call as a row under the session's cycle_id

If a session goes silent for >5 minutes without re-registering, its status moves from ALIVE → STALE → DEAD on the dashboard — that's the cue to either re-register or call end_session.

## What to add later (deferred)

- **Auto-register on Claude Code session start.** Currently the agent has to remember to paste the bootstrap. Could be packaged as a `start_session.bat` or as a CLAUDE.md-level convention so every new session does it automatically.
- **Cross-session work threading.** Currently each session gets its own session_cycle_id. For multi-day projects spanning multiple sessions, James could thread them with a stable `cycle_id` arg to log_work.
- **Streaming for high-priority discoveries.** When Aporia surfaces something genuinely novel (e.g., a verified anti-anchor that contradicts a frontier-model claim), it could also `xadd` to `agora:discoveries` via AgoraClient so it appears in the streams view in real-time, not just at the next 4-hour cycle.
